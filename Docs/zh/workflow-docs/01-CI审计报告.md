# Nexus-Node CI/CD 工作流审计报告

> 审计日期：2026-03-31
> 审计对象：`nexus-node/.github/workflows/` 全部 6 个工作流文件
> 审计视角：第一性原理——**编译一次，测试多次，镜像复用**

---

## 1. 当前工作流概览

| 文件 | 触发条件 | Job 数量 | 用途 |
| ------ | --------- | --------- | ------ |
| `ci.yml` | push/PR → main | **20** | 主门禁：lint → security → tests（~14 个测试门禁）→ coverage → release-go-nogo |
| `devnet-smoke.yml` | push/PR → main | 1 | 4 节点 Docker 集群冒烟测试 |

| `deploy-testnet.yml` | 手动触发 | 2 | Docker 镜像构建 + SSH 部署 |

**总计：每次 PR 推送触发 ~22 个并行 Job。**

## 2. 核心问题：冗余构建

### 2.1 重复环境设置模式

`ci.yml` 中全部 20 个 Job 均包含以下 **完全相同** 的前置步骤：

```yaml
# 每个 Job 重复执行（×20）：
- uses: actions/checkout@v6.0.2
- uses: dtolnay/rust-toolchain@...    # 安装 Rust 1.85.0
- uses: Swatinem/rust-cache@v2        # 恢复缓存
- run: sudo apt-get install -y libclang-dev  # 系统依赖
```

`devnet-smoke.yml` 再**额外独立**重复一次完整构建流程。

### 2.3 资源消耗估算

| 指标 | 当前值 | 说明 |
| ------ | ------- | ------ |
| 冷缓存单次编译 | ~12-18 min | Rust 全 workspace + pqcrypto FFI |
| 热缓存单次恢复+验证 | ~3-6 min | 下载 + 解压 + cargo 增量检查 |
| 热缓存总构建开销 | **60-120 min** | 20 Job × 3-6 min 纯缓存开销 |
| 总计费分钟数（热缓存） | **150-300 min** | 包含测试运行时间 |
| Docker 完整构建 | ~15-20 min | Dockerfile 全量编译（ci.yml + devnet-smoke.yml 各一次） |

**每次 PR 的总 runner 时间约 150-300 计费分钟，其中 40-50% 是纯重复编译开销。**

## 3. 架构级问题分析

### 3.1 违反 DRY 原则

| 问题 | 影响 |
| ------ | ------ |
| 20 个 Job 各自独立安装 Rust 工具链 | 每 Job +1-2 min |
| 20 个 Job 各自恢复 Rust 缓存 | 每 Job +2-4 min（缓存竞争/被驱逐） |
| 20 个 Job 各自安装系统依赖 | 每 Job +0.5 min |
| 10+ 个 Job 各自安装 cargo 工具（nextest/audit/llvm-cov） | 每 Job +1-3 min |

### 3.2 `move-vm-smoke` 与 `devnet-smoke.yml` 功能重叠

两者都执行：

1. 安装 Rust → 编译 release 二进制 → 构建 Docker 镜像 → 启动 devnet → 冒烟测试

**区别仅在于：**

- `devnet-smoke.yml`：4 节点，纯基础设施冒烟
- `move-vm-smoke`（ci.yml）：7 节点，额外运行合约冒烟

这完全可以合并为一个 Job，分步执行。

### 3.3 门禁依赖链不合理

```text
当前依赖图（ci.yml）:
                        ┌→ test-vm ──────────────────────────────────┐
                        ├→ correctness-negative ─────────────────────┤
                        ├→ startup-readiness ────────────────────────┤
                        ├→ epoch-reconfig ───────────────────────────┤
lint ──┐                ├→ stake-weighted-quorum ────────────────────┤
       │   test ────────├→ proof-surface ────────────────────────────┤
security ─┘             ├→ recovery ─────────────────────────────────┤
                        ├→ cold-restart-recovery ────────────────────┤
                        ├→ capacity-curves ──────────────────────────┤
                        ├→ config-drift (不需要编译!) ────────────────┤
                        ├→ gas-calibration ──────────────────────────┤
                        ├→ formal-verification ──────────────────────┤
                        ├→ coverage ─────────────────────────────────┤
                        └→ move-vm-smoke (needs test + test-vm + check) ──→ release-go-nogo
```

**问题：** 所有需要 `needs: [test]` 的 Job 都等待 test 完成后才开始，然后各自重新编译。test 的编译产物完全浪费了。

### 3.4 缓存争用与失效

20 个并行 Job 同时读写 `Swatinem/rust-cache`，导致：

- 缓存键冲突（所有 Job 使用相同 key pattern）
- 缓存大小膨胀（每 Job 保存独立的 target/）
- GitHub cache 总量上限 10GB，容易被驱逐

### 3.5 工具重复安装

| 工具 | 安装位置 | 次数/PR |
| ------ | --------- | --------- |
| cargo-nextest | test, correctness-negative | 2 |
| cargo-audit | security, release-go-nogo | 2 |
| cargo-deny | security | 1 |
| cargo-machete | lint | 1 |
| cargo-llvm-cov | coverage | 1 |
| cargo-fuzz | fuzz | 1 |

## 4. 安全审计结论

### 4.1 良好实践 ✅

- 所有 action 使用 SHA pinning（抗 supply chain 攻击）
- `permissions: contents: read` 最小权限
- `concurrency` 配置避免并发冲突
- RUSTSEC 忽略项有文档记录
- Docker 镜像使用非 root 用户
- Healthcheck 配置完善

### 4.2 改进空间 ⚠️

| 项目 | 风险等级 | 说明 |
| ------ | --------- | ------ |
| cargo-audit `--ignore` 项 | 低 | 3 个 RUSTSEC 忽略，需定期复查 |
| `deploy-testnet.yml` SSH key | 中 | 需确保 key rotation 策略 |
| Docker GHCR token 权限范围 | 低 | 应限制为 `packages: write` |

## 5. 改进方向总结

| # | 改进项 | 预期收益 |
| --- | ------- | --------- |
| 1 | **单次构建 + artifact 分发** | 消除 ~18 次重复编译，节省 60-120 min |
| 2 | **cargo nextest archive** | 测试 Job 零编译，下载 archive 即运行 |
| 3 | **合并 devnet-smoke + move-vm-smoke** | 消除 1 次完整 release 编译 + Docker 构建 |
| 4 | **Docker runtime-prebuilt 复用** | 编译产物直接打包，避免 Docker 内重复编译 |
| 5 | **合并细粒度测试 Job** | 减少 Job 数量从 20 → ~8，降低调度开销 |
| 6 | **Pre-flight 门禁前置并行** | fmt/security/drift 不依赖编译，可立即运行 |
| 7 | **工具预构建或缓存** | 使用 cargo-binstall 或预编译二进制 |

**预期优化效果：**

- 总计费分钟：从 ~200 min 降至 ~60-80 min（节约 60-70%）
- 最长流水线时间：从 ~50 min 降至 ~25-30 min
- 每月 runner 成本：按日均 5 次 PR 计，节约 ~2000 分钟/月
